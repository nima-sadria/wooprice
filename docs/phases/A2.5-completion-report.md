## Architecture Changes

### New Tables (migration a2_004, down_revision=a2_003)

| Table | Description |
|---|---|
| `a2_change_sets` | Header record: state, destination_channel, scope, source_snapshot_id |
| `a2_change_set_revisions` | Immutable version snapshot: revision_number, parent_revision_id, digest (unique) |
| `a2_change_set_items` | Per-product row: product_id, proposal_id, proposal_hash, safety_result_id, rule_version_id, proposed_price (Numeric 14,4), current_price, delta |

### Key Design Decisions

- **Digest algorithm**: SHA-256 over `{destination_channel, scope, source_snapshot_id, items[sorted by product_id+proposal_id]}`. Each item includes `product_id`, `proposal_hash`, `safety_result_id`, `rule_version_id`. UUIDs and timestamps excluded â€” identical logical inputs always produce the same digest.
- **Immutability**: `ChangeSetRepository` has no `update_revision` or `delete` methods. Revisions are write-once.
- **State machine** (validated by `ChangeSetRepository.transition_state`):
  - DRAFT â†’ READY, ARCHIVED
  - READY â†’ SUPERSEDED, ARCHIVED
  - SUPERSEDED â†’ ARCHIVED
  - ARCHIVED â†’ (terminal; no outbound transitions)
- **Phase independence**: `proposal_id`, `safety_result_id`, `rule_version_id` stored as plain strings â€” no FK to A2.3/A2.4 tables. Consistent with the A2.3/A2.4 precedent.

|---|---|
| TD-A2.5-001 | `ChangeSet.revisions` cascade="all, delete-orphan" combined with RESTRICT FK on `parent_revision_id` creates an ordering problem if ChangeSet deletion is ever added (revision 2 would block deletion of revision 1). Not reachable: repository has no delete method. |
| TD-A2.5-002 | `ChangeSet.scope` is String(512) with no length validation at service layer. Scope is controlled input; not a risk in current usage. |

| TD-A2.5-004 | `proposal_id`, `safety_result_id`, `rule_version_id` in ChangeSetItem are plain strings with no FK enforcement. Phase independence preserved by design; FKs can be added when integration is approved. |
