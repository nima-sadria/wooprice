# A2.8 — Scheduling Engine

## Phase

A2.8

## Name

Scheduling Engine

## Status

NOT STARTED

## Mission

Execute confirmed, immutable Change Sets at a designated future time. The Scheduling Engine is a time-based triggering layer on top of the Execution Engine. It never bypasses seller confirmation, never re-evaluates rules or safety policies, and never modifies the Change Set it is scheduled to execute.

---

## Scope

- Scheduler model (scheduled execution records: Change Set ID, confirmed digest, scheduled time, owner)
- Lease ownership (single-executor guarantee: only one worker may hold the lease for a scheduled job)
- Heartbeat mechanism (active worker sends heartbeats; missed heartbeats trigger lease release)
- Retry and backoff scheduling (failed scheduled executions retry with configurable backoff)
- Concurrency control (prevent simultaneous execution of the same scheduled job by multiple workers)
- Cancellation support (a scheduled job may be cancelled before its execution window; requires authorized user)
- Recovery from missed schedules (detect and handle jobs whose scheduled time passed without execution)
- Scheduler audit log (per-job record: scheduled time, execution time, outcome, lease holder)
- Scheduler state machine (scheduled → executing → completed | failed | cancelled | missed)

## Out of Scope

- Transformation Rule evaluation (belongs to A2.3)
- Safety Policy evaluation (belongs to A2.4)
- Change Set creation (belongs to A2.5)
- Dry Run execution (belongs to A2.6)
- Seller confirmation (belongs to A2.6)
- Source data collection (belongs to A2.2)
- AI Foundation
- Any modification to existing Workspace scheduling or sync logic

---

## Architecture Rules

- Scheduling never bypasses seller confirmation. A scheduled job must always reference an already-confirmed Change Set with a valid seller confirmation and digest.
- Scheduled execution always executes an already-confirmed, immutable Change Set. The scheduler does not evaluate rules, safety policies, or compute new proposals.
- Lease ownership must be enforced: only one worker may hold the active lease for any scheduled job at any given time.
- A missed heartbeat must trigger lease release and allow another worker to claim the lease.
- Cancellation is only permitted before the execution window begins; a job in `executing` state may not be cancelled.
- Scheduled execution must pass through the same Execution Engine freshness verification as any direct execution (A2.7 rules apply).
- The Scheduling Engine must never modify the Change Set it is scheduled to execute.

## Database / Migration Notes

- New A2 PostgreSQL migration required for: scheduler tables, lease tables, heartbeat tables, scheduler audit log tables.
- Migration must be additive; no existing A2 or SQLite tables may be modified.
- SQLite (`wooprice.db`) not modified.
- Default production stack not modified.

---

## Testing Requirements

- Lease ownership tests: only one worker may hold the lease; concurrent attempts are rejected
- Heartbeat tests: missed heartbeats trigger lease release; active heartbeats prevent takeover
- Cancellation tests: pre-execution cancellation succeeds; mid-execution cancellation is rejected
- Recovery tests: missed schedule detection and handling
- Seller confirmation bypass tests: scheduler must reject any job without a valid seller confirmation
- Concurrency tests: simultaneous job execution attempts by multiple workers are prevented
- Retry/backoff tests: failed jobs retry with correct backoff intervals
- Audit log tests: all job lifecycle events are recorded
- Integration tests against test PostgreSQL instance
- All existing application tests (`pytest`) must continue to pass unchanged
