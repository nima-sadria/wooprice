# A2.8 — Scheduling Engine

## Phase

A2.8

## Name

Scheduling Engine

## Status

CLOSED

## Mission

Enable deferred execution of an already-confirmed, immutable Change Set through the
A2.7 Execution Engine foundation. The Scheduling Engine is a time-based triggering layer
only. It never bypasses seller confirmation, never re-evaluates rules or safety policies,
and never modifies the Change Set it is scheduled to execute.

**A2.8 scope constraint**: No real WooCommerce write APIs connected. No
ChannelExecutionAdapter implementation in A2.8 production code. No existing Workspace
or existing Apply workflow modified.

## Architecture Rules

- Scheduling never authorizes execution by itself. A scheduled run may only dispatch
  execution if the target Change Set is already immutable, confirmed, Dry Run verified,
  and digest-bound.
- A2.7 independently validates all execution prerequisites on every dispatch:
  confirmation digest, Change Set digest, Dry Run state, freshness, idempotency.
  Scheduling cannot override or bypass A2.7 validation.
- Lease ownership must be enforced: only one worker may hold the active lease for any
  run at any given time.
- Expired leases may be reclaimed by updating the existing lease record.
- Heartbeats extend the active lease expiry.
- Cancellation marks PENDING runs as CANCELLED. Terminal-state runs are not modified.
- Retry/backoff is represented as state/data only — no background retry worker.
- Phase independence: change_set_id, confirmation_id, dry_run_id, and execution_id are
  stored as plain strings — no foreign keys to prior-phase tables.
- A2.8 must not contain ChannelExecutionAdapter implementations.
