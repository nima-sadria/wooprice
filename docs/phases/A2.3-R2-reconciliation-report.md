## 1. Context

During A2.3 development, a divergent REMOTE implementation was discovered at commit `7b7b0eb`
that had a different — and architecturally superior — formula engine design than the LOCAL
implementation (commit `0e6f82b`). Both shared the same Alembic revision ID `a2_002`, making
direct merge impossible without a reconciliation step.

**Owner Decision (2026-06-25):** Option 3 — Reconcile into A2.3-R2.
- Engine design from REMOTE: AST-based sandboxed evaluator, 5 rule types, propose()/propose_all() API
- Audit/Provenance from LOCAL: a2_ table prefix, provenance table, execution trace table, publish_version() immutability guard
- New non-conflicting migration revision: `a2_002_r2`
- Hash must exclude UUID and timestamp (determinism requirement)
- Price precision: Numeric(14,4) — resolves TD-A2.3-001 from original A2.3

## 4. Migration Conflict Resolution

**Problem:** Both LOCAL (commit `0e6f82b`) and REMOTE (commit `7b7b0eb`) used `revision = "a2_002"`.
Alembic cannot process two revisions with the same ID.

**Resolution:**
1. Deleted the LOCAL `a2_002` migration file entirely.
2. Created `a2_002_r2` as the canonical reconciled revision.
3. Updated `a2_003_safety_policy_engine.py` `down_revision` from `"a2_002"` to `"a2_002_r2"`.

**Resulting chain:** `a2_000 → a2_001 → a2_002_r2 → a2_003 → head`

## 6. Engine Design

```
RuleEngine.propose(rules, canonical_product_id, source_id, snapshot_id, input_values, currency, generated_at)
  → ProposalEnvelope(proposal: PriceProposal, trace: [TraceStep × 3])

RuleEngine.propose_all(...)
  → list[ProposalEnvelope]  # one per applicable rule, sorted by priority

Evaluation order:
  1. Sort rules by (priority ASC, rule_id ASC) — deterministic tiebreaker
  2. Skip rule if any required_input not in input_values
  3. For first applicable rule → _build_envelope()

_build_envelope():
  TraceStep(1, "input_capture",     input_keys, captured_values, "(captured from caller)")
  TraceStep(2, "formula_evaluation", inputs, {"proposed_price": result}, formula_string)
  TraceStep(3, "hash_computation",  ids/version/currency, {"proposal_hash": hash}, "SHA256(…)")

Hash: SHA256(canonical_product_id, rule_id, rule_version_id, rule_version, proposed_price, currency, input_values)
  — excludes proposal_id (UUID) and generated_at (timestamp)
  — uses sort_keys=True, sorted(input_values.items()) for determinism
```

## 7. Repository Design

**RuleRepository:**
- `create_rule()` / `get_rule()` / `list_active_rules()` / `deactivate_rule()`
- `create_version()` — auto-increments version_number (max+1 pattern)
- `publish_version()` — sets is_published=True; raises ValueError if already published
- `set_current_version()` — switches is_current among published versions; returns False for draft/missing
- `load_active_definitions()` — returns `list[RuleDefinition]` for active rules with current published version

**ProposalRepository:**
- `save(envelope)` — persists PriceProposalRecord → ProposalProvenanceRecord → N ExecutionTraceRecord in one flush
- `find_by_hash(hash)` — deduplication lookup; returns PriceProposal if exists, None otherwise
- `get(proposal_id)` — eager-loads provenance and trace via joinedload
- `list_by_product(canonical_product_id)` — returns all proposals for a product, desc by generated_at
- `list_by_snapshot(snapshot_id)` — returns all proposals for a snapshot
