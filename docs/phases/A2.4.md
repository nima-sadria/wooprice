# A2.4 — Safety Policy Engine

## Phase

A2.4

## Name

Safety Policy Engine

## Status

NOT STARTED

## Mission

Evaluate Price Proposals against configurable safety policies before any Change Set is created or any execution is triggered. The Safety Policy Engine is a read-only evaluation gate; it produces structured safety results but never modifies prices, creates change sets, or writes to any destination.

---

## Scope

- Safety Policy model (definition, scope, thresholds, mode)
- Policy evaluation engine (evaluates a Price Proposal against active policies)
- Configurable policy evaluation (policies are data-driven, not hardcoded)
- Default installation policy: **WARN** (non-blocking by default)
- Policy modes:
  - `WARN` — flag for review; does not block
  - `BLOCK` — hard-block the proposal; Change Set cannot be created
  - `REQUIRE OVERRIDE` — block unless an authorized user explicitly overrides
- Percentage change limit policies (detect unusually large price increases or decreases)
- Missing-zero detection (proposed price suspiciously low — possible dropped zero)
- Extra-zero detection (proposed price suspiciously high — possible extra zero)
- Historical price anomaly support (compare proposal against historical price distribution)
- Category-level policies (per product category thresholds)
- Brand-level policies (per brand thresholds)
- User-level policies (per operator permission scope)
- Channel-level policies (per destination channel)
- Override framework (authorized user can override REQUIRE OVERRIDE with audit trail)
- Structured `SafetyResult` output (audit-ready, includes policy name, mode, triggered threshold, outcome)

## Out of Scope

- Change Set generation
- Dry Run Engine
- Apply Workflow
- WooCommerce write operations
- Scheduling Engine
- AI-driven override decisions
- Autonomous policy modification
- Any modification to existing Workspace safety thresholds or alarm settings

---

## Architecture Rules

- The Safety Policy Engine is a read-only evaluation gate; it never creates, modifies, or applies Change Sets.
- `SafetyResult` must be fully audit-ready: every result must include the policy name, triggered threshold, evaluated value, and outcome.
- Default installation policy mode is `WARN`; no phase may change the default to `BLOCK` or `REQUIRE OVERRIDE` without explicit Owner approval.
- Policy definitions are immutable once published; a new policy version must be created for any threshold change.
- Override decisions must be logged with the authorizing user, timestamp, and justification.
- The Safety Policy Engine must not depend on or call any component from A2.5 (Change Set) or later phases.

- Override framework with audit trail

## Database / Migration Notes

- New A2 PostgreSQL migration required for: safety policy tables, policy version tables, safety result tables, override log tables.
- Migration must be additive; no existing A2 or SQLite tables may be modified.
- SQLite (`wooprice.db`) not modified.
- Default production stack not modified.
