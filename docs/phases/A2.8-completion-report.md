## Known Risks

- **No real adapter**: A2.8 dispatches to DummyExecutionAdapter in tests. When a real adapter is wired in (future phase), freshness verification and idempotency contracts must be re-verified against that adapter's behavior.
- **confirmation_is_valid hardcoded**: If a confirmation is revoked after schedule creation, A2.8 cannot detect this. A2.7 still validates digest binding, Dry Run state, freshness, and idempotency.
- **Retry state only**: No background retry worker. The caller must re-poll `list_due_schedules()` and dispatch new runs after `next_run_at` is reached.
- **Lease reclaim is not truly atomic**: In production under concurrent load, the check-then-insert pattern may allow two workers to reclaim simultaneously. A production-grade implementation should use `SELECT FOR UPDATE` or database-native advisory locks.
