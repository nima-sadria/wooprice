# A2.3 — Transformation Rule Engine

## Phase

A2.3

## Name

Transformation Rule Engine

## Status

CLOSED

## Mission

Transform approved source data into deterministic, reproducible, and fully explainable Price Proposals. The Rule Engine is the single authoritative component responsible for converting raw source inputs into structured proposals. It does not apply prices, validate safety, compute change sets, or write to any destination.

---

## Scope

- Rule Engine architecture and interface
- Rule Definition models (types, parameters, priority, versioning)
- Rule Precedence system (order of rule evaluation)
- Rule Versioning (immutable rule versions; audit-ready history)
- Formula Engine — primary model: **Cost + Profit → Proposed Sell Price**
- Future-ready model: Competitor Price + Competition Rule → Proposed Sell Price (input support only)
- `PriceProposal` model (structured output of rule evaluation)
- Proposal provenance tracking (which rules, which inputs, which version produced this proposal)
- Rule repository (persistence of rule definitions and versions)
- Proposal repository (persistence of price proposals)
- Rule execution trace (per-proposal audit record of rule evaluation steps)
- Deterministic execution guarantee (identical inputs always produce identical proposals)
- Explainable proposal generation (every proposal fully derivable from stored provenance)
- A2 PostgreSQL migration for rule and proposal tables

## Out of Scope

- Apply Workflow
- Dry Run Engine
- Safety Policy Engine
- Change Set Engine
- WooCommerce write operations
- Execution Engine
- Scheduling Engine
- AI recommendations
- Competitor price scraping
- Competitor APIs
- External market data collection
- Any modification to existing Workspace, Preview Engine, or production Apply path

## Architecture Rules

- Rule Engine outputs Price Proposals only. A Price Proposal is **not** a Final Applied Price.
- Identical inputs must always produce identical Price Proposals (determinism guarantee).
- Rule execution must be deterministic; no random, time-dependent, or external-query behavior is permitted during rule evaluation.
- Proposal generation must be fully reproducible from stored provenance; given the stored rule version and stored source snapshot, any proposal must be re-derivable at any future time.
- Competitor price is an input only. The Rule Engine may accept competitor price as a data input when provided; it must never collect, scrape, or fetch competitor prices autonomously.
- Rule versions are immutable once published; a new rule version must be created for any change.
- No rule evaluation may write to WooCommerce or any external destination.

## Database / Migration Notes

- New A2 PostgreSQL migration required for: rule definition tables, rule version tables, price proposal tables, proposal provenance tables, rule execution trace tables.
- Migration must be additive; no existing A2 or SQLite tables may be modified.
- SQLite (`wooprice.db`) not modified.
- Default production stack not modified.

- Unit tests for all rule types and formula evaluations
- Determinism tests: same inputs → same outputs across multiple runs and environments
- Reproducibility tests: proposals re-derivable from stored provenance
- Rule versioning tests: rule mutations create new versions; old versions remain valid
- Rule precedence tests: correct evaluation order for overlapping rules
- Proposal provenance tests: every proposal traceable to its rule version and source snapshot
- Integration tests against test PostgreSQL instance

## Owner Decisions

- **Competitor price is an input only.** The Rule Engine may accept competitor price as a data input when provided by an approved adapter. The Rule Engine must never autonomously fetch, scrape, or collect competitor prices.
