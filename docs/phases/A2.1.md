# A2.1 — Canonical Product Model + PostgreSQL Foundation

## Phase

A2.1

## Name

Canonical Product Model + PostgreSQL Foundation

## Status

CLOSED

## Mission

Create the foundational A2 platform model and PostgreSQL foundation on which all subsequent A2 phases are built. Establish the normalized product schema, migration tooling, and infrastructure isolation that allow the A2 track to develop independently of the existing production FastAPI application.

---

## Scope

- PostgreSQL service definition (`docker-compose.a2.yml` compose override)
- A2 migration tooling (`alembic_a2/` framework separate from the existing `alembic/`)
- Canonical Product model and PostgreSQL schema
- Channel Listing model
- Channel Credential model
- Repository layer (CRUD operations for all A2 models)
- Service layer (business logic wrapping repositories)
- Migration strategy documentation
- All above components fully isolated from the existing FastAPI application

## Out of Scope

- Source adapters
- Transformation Rule Engine
- Safety Policy Engine
- Change Set Engine
- Dry Run Engine
- Execution Engine
- Scheduling Engine
- AI Foundation
- Production cutover
- Any modification to the existing SQLite database (`wooprice.db`)
- Any modification to the existing FastAPI application (`app/`)

---

## Architecture Rules

- PostgreSQL schema is A2-only; the SQLite database (`wooprice.db`) remains the production store for the existing application and must not be modified by any A2 phase.
- `docker-compose.a2.yml` must compose additively; it must never be referenced by or required by `docker-compose.yml`.
- No A2 code may be loaded, imported, or executed by the existing FastAPI application.
- The default production stack (`docker compose up -d`) must remain unaffected by all A2 work.
- A2 migration tooling (`alembic_a2/`) is separate from the existing migration tooling (`alembic/`) and must never modify tables owned by the existing application.

## Production Safety Rules

**Global A2 Rules — apply to every A2 phase:**

- A2 must remain additive until reconciliation and parity verification pass.
- Every phase must remain independently auditable.
- Each phase must preserve backward compatibility with the existing Workspace until production cutover is explicitly approved.
- No phase may introduce functionality belonging to future phases.
- No production cutover without explicit Owner approval.
- No production deployment during A2 development unless explicitly approved by the Owner.
- No Apply integration outside its approved phase.
- No Dry Run integration outside its approved phase.
- No Safety Engine integration outside its approved phase.
- No Change Set logic outside its approved phase.
- No WooCommerce write logic outside its approved phase.
- Protected systems always require explicit Owner approval before modification.

**Protected Systems — require explicit Owner approval before modification:**

- Workspace UI
- Preview Engine
- Existing Cache Synchronization
- Apply Workflow
- Dry Run Workflow
- Emergency Apply
- JWT/Auth
- Pricing Validation
- WooCommerce Write Path
- Database Migrations
- Maintenance Mode
- Production Deployment
- Rollback / Undo

---

## Owner Decisions

- A2 PostgreSQL services must be isolated in a compose override file; the default production stack must remain unmodified.
- SQLite (`wooprice.db`) must not be modified by any A2 phase until an explicit migration phase is approved.
