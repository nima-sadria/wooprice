## Mission

Create immutable, versioned Change Sets from approved Price Proposals that have passed Safety Policy evaluation. A Change Set is the single authoritative record of what is proposed to change, for which products, on which channels, based on which proposals and safety results. Once created, a Change Set is never modified; any revision creates a new version.

---

## Scope

- `ChangeSet` model (header: scope, channel, proposal binding, safety binding, state)
- `ChangeSetItem` model (per-product row: product ID, proposed value, current value, delta)
- Immutable revision system (no Change Set may be mutated after creation)
- Change Set version history (full audit trail of all versions for a given scope)
- `Revision` model (metadata for each version: creator, timestamp, digest, reason)
- Digest generation (cryptographic hash of all Change Set items and bindings)
- Digest invalidation rules (conditions that invalidate a Change Set and require a new version)
- Source provenance binding (Change Set bound to the specific source snapshot it was derived from)
- Rule provenance binding (Change Set bound to the specific rule versions used)
- Safety binding (Change Set bound to the specific safety result that cleared it)
- `ChangeSet` state machine (draft → approved → executing → applied | invalidated | cancelled)
- A2 PostgreSQL migration for Change Set and revision tables

## Architecture Rules

- A Change Set is immutable once created; no field may be modified after the Change Set is committed to the repository.
- Any modification to scope, proposals, safety results, or channel requires creation of a new Change Set version; the old version is retained and marked superseded.
- The Change Set digest must be derived from all items and all bindings; any change to any item or binding must produce a different digest.
- The Dry Run Engine (A2.6) and Execution Engine (A2.7) must both verify the Change Set digest before operating; a digest mismatch is a hard block.
- Change Set state transitions must be enforced by the state machine; no state may be skipped or reversed except by explicit cancellation or invalidation.
- A Change Set in `executing` state may not be cancelled; only `draft` and `approved` Change Sets may be cancelled.

## Testing Requirements

- Immutability tests: any attempt to modify a committed Change Set must fail
- Digest correctness tests: identical inputs produce identical digests; any item change produces a different digest
- Digest invalidation tests: all defined invalidation conditions produce invalidated Change Sets
- State machine tests: all valid transitions pass; all invalid transitions are rejected
- Provenance binding tests: source, rule, and safety bindings are correctly stored and retrievable
- Version history tests: superseded versions are retained and queryable
- Cancellation tests: only `draft` and `approved` Change Sets may be cancelled
- Integration tests against test PostgreSQL instance
- All existing application tests (`pytest`) must continue to pass unchanged
