# A2.7 — Execution Engine

## Phase

A2.7

## Name

Execution Engine

## Status

NOT STARTED

## Mission

Execute approved, confirmed, immutable Change Sets by writing to destination channels. The Execution Engine is the only A2 component authorized to write to WooCommerce or any external destination. It must verify live destination freshness immediately before every write, enforce idempotency, and support full recovery from partial execution.

---

## Scope

- Execution batch management (group Change Set items into efficient destination batches)
- Adapter batching (per-destination batching strategy, respecting adapter capabilities)
- Retry logic with exponential backoff (for transient destination failures)
- Idempotency enforcement (re-executing a batch must produce the same outcome as the first execution)
- Freshness verification (live destination state check immediately before each write)
- Recovery from partial execution (resume a batch after interruption without re-executing completed items)
- Duplicate execution prevention (a confirmed Change Set may only be executed once; re-execution requires explicit Owner approval)
- Execution audit log (per-item record of: attempt, outcome, destination response, timestamp)
- Execution state machine (pending → executing → completed | failed | partial)

## Out of Scope

- Scheduling Engine (belongs to A2.8)
- AI Foundation
- Safety Policy evaluation (belongs to A2.4)
- Transformation Rule evaluation (belongs to A2.3)
- Source data collection (belongs to A2.2)
- Change Set creation (belongs to A2.5)
- Dry Run execution (belongs to A2.6)
- Seller confirmation (belongs to A2.6)
- Any modification to existing Workspace Apply or WooCommerce write logic

---

## Architecture Rules

- The Execution Engine must verify live destination freshness immediately before writing each item. If the live destination state differs from the state at Change Set creation time, execution for that item must be hard-blocked.
- Failure to verify freshness must hard-block execution for the affected item; it may not be skipped or retried without re-running the Dry Run.
- Idempotency must be enforced at the item level; executing the same item twice must produce the same destination state.
- A confirmed Change Set may be executed only once under normal operation. Re-execution of an already-executed Change Set requires explicit Owner approval and creates a new execution record.
- The Execution Engine is the only A2 component authorized to write to WooCommerce or any destination channel; no other A2 component may perform destination writes.
- Every destination write must be preceded by a verified seller confirmation with a matching Change Set digest.
- Partial execution must be recoverable; the engine must be able to resume from the last successfully executed item.

- Audit log repository

---

## Database / Migration Notes

- New A2 PostgreSQL migration required for: execution tables, execution item tables, audit log tables, execution state tables.
- Migration must be additive; no existing A2 or SQLite tables may be modified.
- SQLite (`wooprice.db`) not modified.
- Default production stack not modified.

---

## Testing Requirements

- Freshness verification tests: stale destination state hard-blocks the item write
- Freshness failure tests: failure to verify freshness hard-blocks execution
- Idempotency tests: executing the same batch twice produces the same destination state
- Recovery tests: partial execution can be resumed from the last completed item
- Duplicate execution prevention tests: re-executing an already-executed Change Set is rejected
- Retry tests: transient failures trigger retry with backoff; non-transient failures do not retry
- Audit log tests: every item attempt and outcome is recorded
- State machine tests: all valid transitions pass; all invalid transitions are rejected
- Integration tests against test PostgreSQL instance (and mock destination)
- All existing application tests (`pytest`) must continue to pass unchanged
