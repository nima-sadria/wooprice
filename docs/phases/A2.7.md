# A2.7 — Execution Engine

## Phase

A2.7

## Name

Execution Engine

## Status

CLOSED

## Mission

Execute approved, confirmed, immutable Change Sets through a controlled adapter interface. The Execution Engine enforces five sequential prerequisites (valid confirmation, digest binding, Dry Run result, digest verification, independent digest recomputation), verifies live destination freshness per item immediately before execution, enforces idempotency, and supports recovery detection from stale RUNNING states.

**A2.7 scope constraint**: Real WooCommerce write APIs are NOT connected in this phase. All execution goes through DummyExecutionAdapter (simulation only). No existing Workspace Apply or WooCommerce write logic is modified.

## Scope

- Execution model with state machine (PENDING → RUNNING → SUCCEEDED | FAILED | BLOCKED | CANCELLED)
- Execution batch model (groups items into a single batch per execution)
- Execution item model with state machine (PENDING → RUNNING → SUCCEEDED | FAILED | BLOCKED | SKIPPED)
- Execution attempt model (immutable record of each adapter call per item)
- Prerequisite validation (5-stage hard block chain before any item is processed)
- Freshness verification contract (ChannelExecutionAdapter.verify_freshness — hard block on failure)
- Adapter execution interface (ChannelExecutionAdapter ABC)
- DummyExecutionAdapter (simulates success, freshness failure, transient failure, permanent failure, idempotent retry — no network calls)
- Idempotency enforcement (unique idempotency_key per execution and per item)
- Retry logic (configurable max_attempts; transient failures retry, permanent failures do not)
- Recovery foundation: detection of stale RUNNING executions and items (no automatic recovery)
- ExecutionService: orchestrates full execution lifecycle
- ExecutionRepository: persistence and state machine enforcement

## Architecture Rules

- Execution may only run against: immutable Change Set revision / valid Change Set digest / valid SellerConfirmation / passing Dry Run verification / successful live destination freshness verification. Failure in any of these hard-blocks execution.
- Freshness failure hard-blocks the item (BLOCKED) and the overall execution (BLOCKED).
- Retry is permitted only for transient failures; permanent failures set item to FAILED immediately.
- Idempotency key on execution prevents duplicate execution records; idempotency key on item prevents duplicate item records.
- Phase independence: change_set_id, confirmation_id stored as plain strings — no foreign keys to prior-phase tables.
- Terminal states (SUCCEEDED, FAILED, BLOCKED, CANCELLED for executions; SUCCEEDED, FAILED, BLOCKED, SKIPPED for items) are immutable.
- DummyExecutionAdapter makes no network calls and performs no real writes.
