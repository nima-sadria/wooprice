# A2.2 — Source Adapter Framework

## Phase

A2.2

## Name

Source Adapter Framework

## Status

CLOSED

## Mission

Create the abstraction layer between WooPrice and every future source system. Define the canonical interface through which all source data enters the A2 platform, ensuring that the Rule Engine, Safety Engine, Change Set Engine, and Execution Engine never depend on source-specific logic.

---

## Scope

- `SourceAdapter` abstract interface
- `SourceCapabilities` declaration model
- `SourceRegistry` for adapter registration and management
- `SourceSnapshot` model (point-in-time capture of source data)
- `SourceProvenance` tracking (origin, timing, identity of every snapshot)
- `SourceCheckpoint` persistence (resumable ingestion state)
- `NextcloudSourceAdapter` (XLSX/WebDAV implementation)
- Persistence layer for source data (adapter tables in A2 PostgreSQL)
- Unit and integration tests for all adapter components

## Out of Scope

- Transformation Rule Engine
- Safety Policy Engine
- Change Set Engine
- Dry Run Engine
- Execution Engine
- Scheduling Engine
- AI Foundation
- Product diffing or delta computation
- Mutation planning
- Production cutover

---

## Architecture Rules

- All adapters must be read-only from the A2 perspective; no adapter may write to any source system.
- Adapters must declare their capabilities via `SourceCapabilities`; no capability may be assumed without explicit declaration.
- Checkpoint persistence must be idempotent; replaying a checkpoint must produce the same state as the original ingestion.
- `SourceSnapshot` must be immutable once written; no snapshot may be modified after creation.
- The NextcloudSourceAdapter must not reuse or depend on any existing production WebDAV or spreadsheet code from `app/`.
- No adapter logic may be imported or used by the existing FastAPI application.

## Production Safety Rules

**Global A2 Rules — apply to every A2 phase:**

- A2 must remain additive until reconciliation and parity verification pass.
- Every phase must remain independently auditable.
- Each phase must preserve backward compatibility with the existing Workspace until production cutover is explicitly approved.
- No phase may introduce functionality belonging to future phases.
- No production cutover without explicit Owner approval.
- No production deployment during A2 development unless explicitly approved by the Owner.
- No Apply integration outside its approved phase.
- No Dry Run integration outside its approved phase.
- No Safety Engine integration outside its approved phase.
- No Change Set logic outside its approved phase.
- No WooCommerce write logic outside its approved phase.
- Protected systems always require explicit Owner approval before modification.

**Protected Systems — require explicit Owner approval before modification:**

- Workspace UI
- Preview Engine
- Existing Cache Synchronization
- Apply Workflow
- Dry Run Workflow
- Emergency Apply
- JWT/Auth
- Pricing Validation
- WooCommerce Write Path
- Database Migrations
- Maintenance Mode
- Production Deployment
- Rollback / Undo

---

## Owner Decisions

- Adapters are read-only; no source system may be written to via any A2 adapter.
- The NextcloudSourceAdapter must be implemented independently from existing production spreadsheet code.
