# WooPrice Beta — Security Architecture

**Document:** SECURITY_ARCHITECTURE.md
**Series:** B1 Architecture Blueprint

---

## Overview

WooPrice Beta handles credentials for Nextcloud and WooCommerce test accounts. Even
though these are test accounts (not production), the security model is production-grade:
secrets are never logged, never committed, and never transmitted in plain text over
internal networks.

---

## Threat Model

### In scope

| Threat | Mitigated by |
|---|---|
| Credential exposure via source control | `.gitignore` rules; installer generates secrets; no defaults |
| Secret leakage via logs | Logger sanitizer; secrets never in log fields |
| Unauthorized API access | JWT authentication on all endpoints; short token TTL |
| Session hijacking | httpOnly + Secure + SameSite cookies; in-memory access tokens |
| SQL injection | SQLAlchemy ORM; parameterized queries only; no raw SQL with user input |
| XSS | React (no `dangerouslySetInnerHTML`); CSP header; API returns structured JSON only |
| CSRF | SameSite=Strict cookies; CSRF token on mutation endpoints |
| Broken access control | Per-endpoint permission guards; feature flag gates |
| Plugin privilege escalation | Manifest-declared permissions; no direct DB access from plugins |
| Insecure defaults | No default passwords; secrets are generated; validation on startup |
| Misconfiguration — Production environment connected | Environment guard; BETA_ variable isolation |

### Out of scope (Beta phase)

- Physical security
- Network-level attacks (DDoS, BGP hijacking)
- Hardware compromise
- Zero-day exploits in Docker, Linux kernel, or dependencies

## Secret Management

### Where secrets live

| Secret | Storage | Transmission |
|---|---|---|
| `BETA_POSTGRES_PASSWORD` | `.env` file (mode 600) | Docker internal network only |
| `BETA_JWT_SECRET` | `.env` file (mode 600) | Never transmitted; used server-side only |
| `BETA_REST_API_SECRET` | `.env` file (mode 600) | Transmitted over internal Docker network only |
| `BETA_NEXTCLOUD_PASSWORD` | `.env` file (mode 600) | HTTPS to Nextcloud only |
| `BETA_WOOCOMMERCE_KEY` | `.env` file (mode 600) | HTTPS to WooCommerce only |
| `BETA_WOOCOMMERCE_SECRET` | `.env` file (mode 600) | HTTPS to WooCommerce only |
| Initial admin password | Printed once to terminal; install log (mode 600) | HTTPS form POST |

### What never happens

- Secrets never appear in the managed config TOML file
- Secrets never appear in application logs
- Secrets never appear in API responses
- Secrets never appear in Docker Compose file content (only referenced as `${VARIABLE}`)
- Secrets never appear in source control
- The `.env` file is in `.gitignore` permanently

### Secret strength requirements

| Secret | Minimum strength |
|---|---|
| `BETA_JWT_SECRET` | 64 characters; generated by `openssl rand -base64 64` |
| `BETA_REST_API_SECRET` | 32 bytes hex; generated by `openssl rand -hex 32` |
| `BETA_POSTGRES_PASSWORD` | 32 characters; generated by `openssl rand -base64 32` |
| Admin initial password | 12 characters minimum; entropy-checked |

## Authentication

**Local admin login is mandatory.** Beta authentication must not depend on Nextcloud
or any external identity provider. Local credential login (email + password verified
against the Beta user database) is always available. External identity provider
integrations (Nextcloud, OAuth, LDAP) are optional and their failure must never block
owner or admin access. See Section "Control Plane Resilience" below.

### JWT tokens

```
Access token
  Algorithm: HS256
  Signing key: BETA_JWT_SECRET (in-memory only; loaded from env)
  TTL: BETA_JWT_ACCESS_TTL_MINUTES (default: 15 minutes)
  Payload: { sub: user_id, email, permissions, iat, exp }

Refresh token
  Algorithm: HS256
  Signing key: BETA_JWT_SECRET
  TTL: BETA_JWT_REFRESH_TTL_DAYS (default: 7 days)
  Storage: httpOnly, Secure, SameSite=Strict cookie
  Rotation: new refresh token issued on each silent refresh
```

### Token storage

- **Access token:** in-memory React state only. Never `localStorage`. Lost on tab close
  (silent refresh on next open if refresh cookie still valid).
- **Refresh token:** httpOnly cookie — JavaScript cannot read it; safe from XSS.

### Session invalidation

Sessions are invalidated on:
1. Explicit logout (`DELETE /api/auth/session`)
2. Password reset (all sessions for that user invalidated)
3. Admin deactivation of a user (all sessions invalidated)
4. JWT secret rotation (`wooprice configure rotate --secret jwt`) — all active sessions
   invalidated immediately

## Authorization

### Permission model

Permissions are named strings stored in the Beta database per user. Admin users
have all permissions implicitly. Non-admin users are assigned individual permissions.

Permission check pattern:

```python
from app.beta.auth.dependency import require_permission

@router.post("/execution/apply")
async def apply_change_set(
    _: None = Depends(require_permission("write:execution")),
    ...
):
    ...
```

### Feature flag gates

API endpoints behind feature flags return `HTTP 404` when the flag is disabled —
not `HTTP 403`. This avoids disclosing which features exist but are disabled.

### Plugin permission isolation

Plugin code runs with only the permissions declared in the plugin manifest. The Plugin
Host passes a restricted service interface — plugins cannot call arbitrary service
methods beyond what the interface exposes.

## Transport Security

| Connection | Requirement |
|---|---|
| Browser → Nginx | HTTPS (if `BETA_SSL_MODE != off`); HTTP in `off` mode (lab/testing only) |
| Nginx → App container | HTTP (Docker internal; not accessible externally) |
| App → PostgreSQL | Docker internal; password-authenticated |
| App → Nextcloud | HTTPS required; certificate validated |
| App → WooCommerce | HTTPS required; certificate validated |
| App → Redis | Docker internal; no password needed in Beta (single-node) |

### Content Security Policy (CSP)

The Nginx container sets a strict CSP header for all responses to the frontend:

```
Content-Security-Policy:
    default-src 'self';
    script-src 'self';
    style-src 'self' 'unsafe-inline';
    img-src 'self' data:;
    connect-src 'self';
    font-src 'self';
    frame-ancestors 'none';
    form-action 'self';
```

`unsafe-inline` in `style-src` is a Tailwind CSS constraint; reviewed for removal in B12+.
No `unsafe-eval`. No CDN resources.

### Other security headers (Nginx)

```
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

## Input Validation

All input is validated at the API boundary using Pydantic models. No raw SQL string
construction. All ORM queries use SQLAlchemy's parameterized API.

### Validation layers

1. **Pydantic schemas** — all request bodies and query parameters are typed
2. **SQLAlchemy ORM** — no raw SQL with user-supplied strings
3. **Enum constraints** — status fields, category fields use Python enums
4. **File upload limits** — enforced by Nginx (`client_max_body_size`) and FastAPI
5. **Plugin config validation** — plugin-supplied config validated against JSON Schema

### What is explicitly NOT done

- No `eval()` or `exec()` of user-supplied strings
- No `os.system()` with user-supplied arguments
- No `subprocess` calls with unsanitized input
- No templating with user-supplied strings using `str.format()` on SQL-like patterns

## Audit Log

All security-relevant events are written to `BETA_STORAGE_PATH/logs/audit.log`
(structured JSON, one event per line). Audit events are never deleted automatically.

| Event | Logged fields |
|---|---|
| Login success | user_id, email, ip, timestamp |
| Login failure | email, ip, timestamp, reason |
| Logout | user_id, timestamp |
| Session refresh | user_id, timestamp |
| Password change | user_id, timestamp |
| Permission change | admin_user_id, target_user_id, permission, action, timestamp |
| Feature flag toggle | user_id, flag, previous_value, new_value, timestamp |
| Plugin install | user_id, plugin_id, version, timestamp |
| Plugin enable/disable | user_id, plugin_id, action, timestamp |
| Config change | user_id, key, timestamp (value NOT logged if it's a secret) |
| Secret rotation | user_id, secret_type, timestamp |
| Execution applied | user_id, change_set_id, product_count, timestamp |
| Backup created | user_id, backup_id, path, timestamp |
| Backup restored | user_id, backup_id, timestamp |

## Environment Isolation Guard

The environment guard (`cli/shared/env_guard.py`) runs before every CLI write
operation and every destructive API call:

```
1. Load BETA_ENV from .env
2. If BETA_ENV = "production" without --i-know-what-i-am-doing flag:
   → Print DANGER warning; refuse operation
3. Check BETA_WOOCOMMERCE_URL against Production known-value allowlist:
   → If matched: refuse operation; print PRODUCTION RESOURCE WARNING
4. Check BETA_DATABASE_URL against Production known-value allowlist:
   → If matched: refuse operation; print PRODUCTION RESOURCE WARNING
```

The known-value allowlists are maintained in `cli/shared/env_guard.py` and updated
by the operator after any Production resource change.

## Dependency Security

Python dependencies are pinned in `requirements.txt` with exact version hashes
(generated by `pip-compile --generate-hashes`). No floating version ranges.

Frontend dependencies are pinned in `package-lock.json` with integrity hashes.

Automated dependency vulnerability scanning is part of the B4 CI pipeline (using
`pip-audit` for Python, `npm audit` for Node).

## Control Plane Resilience

**Owner decision — 2026-06-27**

The Control Plane (login, settings, diagnostics, feature flags, plugin manager,
backup/update controls) must remain accessible when one or more Integration Plane
services are down.

**Security implication:** Integration failures must be reported with exact failure
class — not collapsed to "Invalid credentials". The failure classes are:

| Class | Meaning |
|---|---|
| `dns_failure` | DNS resolution failed |
| `tls_failure` | TLS / certificate validation error |
| `timeout` | Connection timed out |
| `unauthorized` | HTTP 401 — wrong or expired credentials |
| `forbidden` | HTTP 403 — credentials valid but access denied |
| `unreachable` | Connection refused / no route to host |
| `invalid_response` | Server replied with unexpected content |

Collapsing a DNS or TLS failure to "Invalid credentials" is a security
anti-pattern: it misleads the operator into rotating credentials when the
real cause is a network or certificate problem.

**Production lesson:** WooPrice 7.5A silently collapsed Nextcloud DNS/TLS
failures to "Invalid Nextcloud credentials". Beta must not repeat this.

**Auth requirement from this decision:** Local admin login (`/api/auth/login`
verified against the Beta user database) must never depend on Nextcloud
connectivity. External auth integrations are optional and must fail safely.

## Incident Response (Beta)

Because this is a Beta environment with test accounts:

1. **Suspected credential compromise:** Run `wooprice configure rotate --all-secrets`
   immediately, then revoke and regenerate WooCommerce and Nextcloud credentials
   in their respective admin panels.
2. **Suspected unauthorized access:** Run `wooprice diagnostics run`, review
   `audit.log`, rotate all secrets.
3. **Data corruption:** Restore from backup using `wooprice backup restore`.
4. **Contact:** Report to Owner immediately. Do not attempt silent remediation.
