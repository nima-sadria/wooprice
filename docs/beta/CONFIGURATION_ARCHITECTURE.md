## Overview

All configuration is sourced from environment variables and managed configuration
files generated by the installer and CLI. No application code reads `os.environ`
directly — all access flows through the Configuration Manager.

---

## Configuration Sources (Priority Order)

```
1. Environment variables           ← highest priority (override everything)
2. Managed config file             ← written by installer/CLI; read by app on startup
3. Database-backed settings        ← runtime-mutable (feature flags, adapter config)
4. Default values (code)           ← lowest priority; always present as fallback
```

Secrets always flow from environment variables (source 1). They must never be stored
in database-backed settings or managed config files (sources 2–3).

---

## Environment Variables

All Beta environment variables use the `BETA_` prefix. They are loaded from the
generated `.env` file by Docker Compose and injected into the container environment.
The `.env` file is never committed — `.gitignore` always excludes it.

### Required variables (installer must collect and generate these)

| Variable | Type | Description |
|---|---|---|
| `BETA_ENV` | `beta` / `dev` / `production` | Environment label; controls safety warnings |
| `BETA_DOMAIN` | string | Hostname where Beta is served |
| `BETA_PORT` | integer | Application server port |
| `BETA_DATABASE_URL` | connection string | Full PostgreSQL URL for A2 database |
| `BETA_POSTGRES_DB` | string | PostgreSQL database name |
| `BETA_POSTGRES_USER` | string | PostgreSQL username |
| `BETA_POSTGRES_PASSWORD` | secret | PostgreSQL password |
| `BETA_JWT_SECRET` | secret | JWT signing key (min 64 chars; generated) |
| `BETA_REST_API_SECRET` | secret | REST API key for internal service calls |
| `BETA_NEXTCLOUD_URL` | URL | Nextcloud base URL for source adapter |
| `BETA_NEXTCLOUD_FILE_PATH` | path | Path to test spreadsheet in Nextcloud |
| `BETA_NEXTCLOUD_USERNAME` | string | Nextcloud username |
| `BETA_NEXTCLOUD_PASSWORD` | secret | Nextcloud password |
| `BETA_WOOCOMMERCE_URL` | URL | WooCommerce test store URL |
| `BETA_WOOCOMMERCE_KEY` | secret | WooCommerce consumer key |
| `BETA_WOOCOMMERCE_SECRET` | secret | WooCommerce consumer secret |
| `BETA_TIMEZONE` | tz string | e.g., `Europe/Amsterdam` |
| `BETA_CURRENCY` | ISO 4217 | e.g., `EUR` |
| `BETA_ADMIN_EMAIL` | email | Initial admin account email |
| `BETA_STORAGE_PATH` | path | Base path for app storage |
| `BETA_BACKUP_PATH` | path | Path for backup archives |
| `BETA_SSL_MODE` | enum | `off` / `self-signed` / `letsencrypt` / `manual` |

### Optional variables (defaults applied if absent)

| Variable | Default | Description |
|---|---|---|
| `BETA_LOG_LEVEL` | `INFO` | Application log level |
| `BETA_JWT_ACCESS_TTL_MINUTES` | `15` | Access token TTL |
| `BETA_JWT_REFRESH_TTL_DAYS` | `7` | Refresh token TTL |
| `BETA_MAX_UPLOAD_MB` | `50` | Max file upload size |
| `BETA_PLUGIN_DIR` | `$BETA_STORAGE_PATH/plugins` | Plugin installation directory |
| `BETA_WORKER_CONCURRENCY` | `2` | Background worker concurrency |
| `BETA_SCHEDULER_POLL_SECONDS` | `30` | Scheduler polling interval |
| `BETA_BACKUP_RETAIN_DAYS` | `30` | Backup retention period |

## Managed Configuration File

The managed configuration file is written by the installer (`install.sh`) and updated
by the CLI (`wooprice configure`). It is a TOML file stored outside the repository at
a path specified by the installer.

**File path:** `$BETA_STORAGE_PATH/config/wooprice-beta.toml`

```toml
# WooPrice Beta — managed configuration file
# Generated by installer or wooprice configure
# DO NOT EDIT MANUALLY unless in emergency. Use: wooprice configure

[meta]
version = "beta-1.0.0"
installed_at = "YYYY-MM-DDTHH:MM:SSZ"
installer_version = "1.0.0"

[app]
env = "BETA_ENV"
domain = "BETA_DOMAIN"
port = 0
timezone = "BETA_TIMEZONE"
currency = "BETA_CURRENCY"
storage_path = "BETA_STORAGE_PATH"
backup_path = "BETA_BACKUP_PATH"
ssl_mode = "BETA_SSL_MODE"
log_level = "INFO"

[database]
# Note: password is NOT stored here — it comes from BETA_POSTGRES_PASSWORD env var
postgres_db = "BETA_POSTGRES_DB"
postgres_user = "BETA_POSTGRES_USER"
postgres_host = "postgres"
postgres_port = 5432

[source]
type = "nextcloud"
nextcloud_url = "BETA_NEXTCLOUD_URL"
nextcloud_file_path = "BETA_NEXTCLOUD_FILE_PATH"
nextcloud_username = "BETA_NEXTCLOUD_USERNAME"
# Password: BETA_NEXTCLOUD_PASSWORD env var

[channel]
woocommerce_url = "BETA_WOOCOMMERCE_URL"
# Key and secret: BETA_WOOCOMMERCE_KEY, BETA_WOOCOMMERCE_SECRET env vars

[features]
# Feature flags are stored in the database; these are the boot-time defaults
# if the database is not yet initialized.
rule_engine = true
safety_engine = true
change_sets = true
dry_run = true
execution = true
scheduler = true
ai = true
multi_channel = false
competitor_features = false
plugin_system = true
```

**Rules for the managed config file:**
- Secrets (passwords, keys, tokens) are never written here — only referenced via env var.
- The file is owned by the application user (not root).
- Write permission is restricted to the application user and the installer script.
- Emergency edits must be followed by `wooprice configure --verify` to detect drift.

## Generated Secrets

All secrets are generated at install time by the installer using `openssl rand`:

```bash
# JWT secret (64 bytes, base64url encoded)
BETA_JWT_SECRET=$(openssl rand -base64 64 | tr -d '\n')

# REST API secret (32 bytes, hex encoded)
BETA_REST_API_SECRET=$(openssl rand -hex 32)

# PostgreSQL password (32 bytes, base64 encoded, URL-safe)
BETA_POSTGRES_PASSWORD=$(openssl rand -base64 32 | tr -d '\n+/=' | head -c 32)
```

Generated secrets are:
- Written to the `.env` file immediately after generation
- Never echoed to the terminal in plain text
- Never logged
- Never stored in source control
- Unique per installation — they are never re-used

## Configuration Validation

The Configuration Manager validates all required variables on application startup.
Missing or invalid values cause startup failure with a clear error message.

### Validation checks

| Variable | Validation |
|---|---|
| `BETA_ENV` | Must be one of: `beta`, `dev`, `production` |
| `BETA_PORT` | Must be integer 1024–65535 |
| `BETA_DATABASE_URL` | Must be a valid PostgreSQL connection string |
| `BETA_JWT_SECRET` | Must be at least 64 characters |
| `BETA_REST_API_SECRET` | Must be at least 32 characters |
| `BETA_NEXTCLOUD_URL` | Must be a valid URL (scheme required) |
| `BETA_WOOCOMMERCE_URL` | Must be a valid URL (scheme required) |
| `BETA_TIMEZONE` | Must be a valid IANA timezone string |
| `BETA_CURRENCY` | Must be a valid ISO 4217 code (3 uppercase letters) |
| `BETA_STORAGE_PATH` | Must exist and be writable |
| `BETA_BACKUP_PATH` | Must exist and be writable |
| `BETA_SSL_MODE` | Must be one of: `off`, `self-signed`, `letsencrypt`, `manual` |

### Startup validation flow

```
Application startup
  ↓
Load .env (via Docker Compose injection — already in environment by this point)
  ↓
ConfigurationManager.validate()
  ├── Check all required variables are present
  ├── Validate types and formats
  ├── Check BETA_ENV is NOT "production" unless explicitly authorized
  └── Check storage paths exist and are writable
  ↓
If validation fails → log error + exit(1) with clear message
  ↓
If validation passes → inject typed Config object into dependency graph
```

## Configuration Lifecycle

```
Install
  │
  ├── installer interactive wizard collects all values
  ├── installer generates secrets
  ├── installer writes .env file
  ├── installer writes managed config file
  └── installer writes Docker Compose file from template
  ↓
Run
  │
  ├── Docker Compose injects .env into container environment
  ├── Configuration Manager loads and validates on startup
  └── Config object available to all services via dependency injection
  ↓
Update config
  │
  ├── wooprice configure [--domain X] [--ssl-mode Y] ...
  ├── CLI updates managed config file
  ├── CLI updates .env for secret rotation
  └── CLI restarts affected services
  ↓
Secret rotation
  │
  ├── wooprice configure --rotate-secret jwt
  ├── New secret generated
  ├── .env updated
  ├── Services restarted
  └── Old tokens invalidated (JWT expiry shortened to 0 temporarily)
```

## CLI-Driven Updates

Configuration changes after installation use `wooprice configure`:

```bash
# Change domain
wooprice configure --domain new.example.com

# Change SSL mode
wooprice configure --ssl-mode letsencrypt

# Rotate JWT secret (invalidates all current sessions)
wooprice configure --rotate-secret jwt

# Verify configuration consistency
wooprice configure --verify

# Show current configuration (secrets redacted)
wooprice configure --show
```

All `wooprice configure` operations:
1. Read the current managed config file
2. Apply the requested change
3. Write the updated managed config file
4. Update `.env` if needed
5. Restart affected services (`docker compose restart <service>`)
6. Run `wooprice health` to confirm the change took effect
7. Write the operation to `BETA_STORAGE_PATH/logs/audit.log`

---

## Emergency-Only Manual Editing

Manual editing of the managed config file or `.env` is **emergency-only**. After any
manual edit, the operator must run:

```bash
wooprice configure --verify
```

This command compares the running configuration with the managed config file, reports
any discrepancies, and optionally restarts services to apply the corrected values.

## Environment Labels

The `BETA_ENV` variable drives environment labels displayed in the UI and CLI. Labels
are never suppressible.

| Value | UI label | CLI banner | Behavior |
|---|---|---|---|
| `beta` | `[BETA]` | `[BETA ENVIRONMENT]` | Normal Beta operation |
| `dev` | `[DEV]` | `[DEVELOPMENT ENVIRONMENT]` | Dev mode; extra debug output |
| `production` | `[PRODUCTION]` | Blocked | CLI refuses destructive commands; prompts for confirmation on every write |

---

## Beta / Dev / Production Separation

| Concern | Beta | Dev | Production |
|---|---|---|---|
| Database | Own `BETA_POSTGRES_DB` | Separate dev DB (seeded from fixtures) | Production DB — never shared |
| JWT secret | Generated at install | Dev-only secret | Production secret — never in Beta |
| Storage path | `BETA_STORAGE_PATH` | Local path on dev machine | Production path — never shared |
| WooCommerce | Test store only | Mock or test store | Real store — never in Beta |
| Nextcloud | Test spreadsheet only | Local test file | Real spreadsheet — never in Beta |
| Log verbosity | INFO | DEBUG | WARN/ERROR |
| Feature flags | Configurable | All enabled by default | Restricted |

**The `BETA_ENV=production` guard:** if the CLI or installer detects `BETA_ENV=production`,
it prints a large warning, requires explicit `--i-know-what-i-am-doing` flag for any
destructive operation, and logs every action to the audit log with a DANGER prefix.

## Control Plane Resilience

**Owner decision — 2026-06-27**

Configuration access is a Control Plane operation. The Configuration Manager and
all configuration CLI commands (`configure show`, `configure verify`, `configure set`)
must operate without any Integration Plane service being available.

**Configuration must never depend on Integration Plane availability:**
- `configure show` reads the local managed config file and environment — no network calls.
- `configure verify` validates the local config against the B3 ConfigValidator — no network calls.
- `configure set` writes the local managed config file — no network calls.
- The Configuration Manager loads and validates on startup from local environment only.

**Integration credentials are configuration values, not live integrations.** Setting or
viewing `BETA_NEXTCLOUD_URL`, `BETA_NEXTCLOUD_PASSWORD`, `BETA_WOOCOMMERCE_KEY`, and
similar variables through `configure set` must work even when the target service is
unreachable. The purpose is precisely to allow the operator to fix bad credentials
during an integration outage.

**Failure class transparency:** When a runtime component tests an integration
(e.g., `wooprice health sources`, `wooprice integrations test nextcloud`), it must
report the exact failure class from the integration response:

| Class | Trigger |
|---|---|
| `dns_failure` | DNS resolution failure on the integration URL |
| `tls_failure` | TLS handshake or certificate validation failure |
| `timeout` | Connection or read timeout |
| `unauthorized` | HTTP 401 response |
| `forbidden` | HTTP 403 response |
| `unreachable` | Connection refused, no route to host |
| `invalid_response` | Non-standard or unexpected server response |

Collapsing any of these to "Invalid credentials" is prohibited. The production
WooPrice 7.5A incident (DNS/TLS failure misreported as "Invalid Nextcloud
credentials") is the explicit motivation for this requirement.
